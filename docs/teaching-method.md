# 先導課程教學方法

這份文件記錄 Day 1 Prep Course 的教學設計意圖。

它不是逐字講稿，也不是學員操作手冊。它的讀者是講師、助教，以及未來維護這份 template 的人。

---

## 課程定位

這個 repo 只服務第一天先導課程。

先導課程的任務不是提前完成後續專案，而是建立一個低風險、可完成、可展示的共同起點。

學生在這兩小時內需要完成的不是「做出厲害作品」，而是走過一次縮小版的工程協作流程：

```text
自然語言描述
→ 結構化資料
→ Coding Agent 修改 repo
→ 人類檢查 diff
→ 前端展示成果
```

這個流程會成為後續課程的共同語言。

---

## 核心教學假設

### 1. 學員需要先建立安全感

先導課程發生在營隊第一天。學員可能還不熟悉隊友，也可能擔心自己程度不夠。

因此第一個 repo 任務必須：

- 容易理解
- 不需要複雜程式能力
- 可以快速看到成果
- 失敗時容易修正
- 不會讓學員暴露能力高低

profile 與 faction 設計的重點就在這裡：它好填、有趣、可公開，也能產生展示效果。

---

### 2. Agentic Coding 的第一步不是寫程式，而是建立可讀上下文

Coding Agent 需要 repo 裡的脈絡才能工作。

所以這堂課不直接要求學員讓 Agent 寫功能，而是先讓學員體驗：

```text
我用自然語言描述偏好
→ Agent 讀懂並轉成 JSON
→ schema 限制資料格式
→ 前端根據資料產生畫面
```

這能讓學員理解：

> AI 不只是聊天工具；在工程專案裡，它會根據檔案、規格、資料與任務說明來修改 repo。

---

### 3. 版本控制的入口應該從「看得見修改」開始

第一天不應把 Git 教成指令記憶。

更重要的是讓學員先建立這個習慣：

```text
Agent 產生修改
→ 我看 diff
→ 我確認它沒有亂改
→ 我才 commit / PR
```

這比一開始理解 branch、merge、rebase 更重要。

---

### 4. 成就感來自公開可見的成果

如果學生只完成帳號登入、工具安裝和指令測試，會覺得這只是前置作業。

因此先導課程需要一個可視化成果：

- GitHub 頭貼出現在頁面上
- 自己的暱稱與一句話介紹出現在卡片上
- 小隊成員被整理成展示頁
- 技術陣營統計會隨著資料增加而變化
- 頁面有輪播、hover、click 等互動

這些效果不是為了炫技，而是為了讓學員感覺：

> 我剛剛真的讓 repo 變了，而且這個變化被網頁展示出來了。

## 文件設計原則

這個 repo 的文件要服務不同讀者，不應混在一起。

### 給 Coding Agent 的文件

代表文件：`AGENTS.md`

目的：讓 Agent 知道工作邊界、資料規格、安全限制與修改原則。

寫法：

- 英文
- 明確
- 可執行
- 不包含教學推演
- 不包含講師與助教的背景討論

---

### 給學員的文件

代表文件：`README.md`、`tasks/`、`notes/README.md`、`profiles/README.md`

目的：讓學生能照著完成任務。

寫法：

- 繁體中文
- 低壓力
- 步驟清楚
- 不要求一次理解整個 repo
- 不過度解釋工具歷史與理論

---

### 給講師與助教的文件

代表文件：`docs/day1-flow.md`、`docs/teaching-method.md`、`docs/privacy.md`、`docs/repo-structure.md`

目的：讓課程工作人員知道為什麼這樣安排，以及卡關時應該如何支援。

寫法：

- 可以保留教學意圖
- 可以說明設計取捨
- 仍然避免記錄私人對談或未整理的推演
- 以可重用的課程設計語言撰寫

## 為什麼用 profile，而不是直接做小專案？

兩小時先導課程的限制很明確：

- 學員剛進入營隊，需要暖機
- 帳號與工具可能需要測通
- 隊友還不熟
- 程度差異可能很大
- 如果一開始就做功能，容易把焦點放在「我會不會寫」

profile 任務比較適合，因為它把焦點放在：

- 我能否描述自己的偏好
- Agent 能否把自然語言轉成結構化資料
- 我能否檢查 Agent 的修改
- 小隊能否共同看到成果

這是 Agentic Coding 的入門，不是正式專案開發。

---

## 為什麼加入 faction？

OS、瀏覽器、編輯器 / IDE 的陣營偏好有幾個教學價值：

1. 好填，不需要自評能力。
2. 有社群梗，能降低課堂壓力。
3. 可統計，能自然引出資料結構與資料清理問題。
4. 能提醒大家：不同工具偏好會影響協作方式。
5. 能把 JSON、schema、前端展示串成一個完整體驗。

但 faction 只能作為趣味元素，不應變成嘲笑或區分強弱的工具。

---

## 為什麼 faction 使用固定代號？

如果讓學員自由填顯示文字，統計會很快失控：

```text
VS Code
vscode
VSCode
Visual Studio Code
vs code
```

因此資料裡使用固定代號：

```json
"editor": "vscode"
```

畫面上再轉成顯示名稱：

```text
VS Code
```

這是一個很適合先導課程的工程觀念：

> 好的系統不是期待所有人永遠填對，而是用資料規格降低錯誤機率。

---

## 講師收束語方向

先導課程最後可以收束成：

```text
今天我們不是在學一堆工具，而是在走一次工程協作的縮小版流程。

你用自然語言提供脈絡，Agent 把它轉成 repo 裡的資料，前端把資料變成畫面，而你負責檢查 diff。

後續課程會變得更複雜，但核心習慣一樣：把想法寫清楚、讓修改可檢查、讓成果可以被團隊看見。
```

---

## 維護時的取捨原則

未來如果要修改這份 template，請優先保持以下原則：

1. **不要把先導課程變成正式專案。** 兩小時內要能完成。
2. **不要讓學員手寫複雜 JSON。** JSON 是 Agent 產物，不是主要作業。
3. **不要增加太多欄位。** profile 要好填，公開風險要低。
4. **不要加入過多依賴。** 前端要可讀、可修、可被 Agent 小範圍修改。
5. **不要犧牲 diff 檢查。** Agent 的輸出必須能被人類審查。
6. **不要把文件寫成對談紀錄。** 保留教學方法，但以整理後的設計原則呈現。
